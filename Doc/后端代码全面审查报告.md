# vlOcr 后端代码全面审查报告

审查时间：2026-06-07

审查范围：`vlmOcr_api` 后端项目，包括 Express 启动配置、路由、中间件、数据库服务、OCR/分类工具、上传下载、第三方 API 封装、SQL 初始化脚本和后端依赖声明。

审查方式：静态代码审查为主，辅以轻量依赖解析命令。未运行 `npm run build`，未做类型检查，未运行 `npm ts:check`。

## 结论摘要

后端已经做了一轮安全加固，例如 JWT 密钥和数据库配置改为启动必填、SQL 查询大多参数化、历史记录按用户隔离、上传文件名随机化、远程 HTTPS URL 做白名单和内网 IP 检查。这些方向是对的。

但当前仍存在若干会影响安全或功能稳定性的 bug。最需要优先处理的是：OCR 接口允许用户传入服务器本地文件路径、业务模型增删改缺少管理员权限、上传文件没有用户归属导致任意登录用户可枚举和删除文件、统计日期校验不完整、历史搜索分页总数不准确、GLM 请求清理文件的时机可能早于请求真正完成。

## P0 / 高危问题

### 1. OCR 接口可读取服务器本地文件路径

严重程度：高

位置：

- `vlmOcr_api/routes/ocr.js:99`
- `vlmOcr_api/routes/ocr.js:54`
- `vlmOcr_api/ocrApi/GLMFlash.js:39`
- `vlmOcr_api/ocrApi/GLMFlash.js:44`
- `vlmOcr_api/ocrApi/GLMFlash.js:77`
- `vlmOcr_api/ocrApi/PaddleOcrVl.js:54`
- `vlmOcr_api/utils/classifier.js:92`

问题说明：

`/ocr/glm-chat`、`/ocr`、`/ocr/paddle-vl`、`/classifyOcr` 等接口会接收 `imgUrl` 或 `imageUrl`。当传入值不是 `http` 开头时，`GLMFlash.resolveImageInput()`、`PaddleOcrVl.resolveInputFile()` 和 `classifier.classifyImage()` 会把它当成本地文件路径处理。

这意味着任意已登录用户可以提交类似 `F:\some\file.jpg`、`..\some\file.pdf` 或部署机器上的绝对路径。只要扩展名符合 JPG、PNG、WEBP、PDF 等检查，后端就可能读取该文件，并把内容发送到第三方 OCR/大模型服务或复制到 `public/images`。

影响：

- 服务器本地图片、PDF 等敏感文件可能被越权读取。
- 本地文件内容可能被上传到第三方 API。
- 路径探测会暴露服务器目录结构和文件是否存在。

建议：

- 外部接口不要接受任意本地路径。
- 只允许使用本系统上传接口返回的 `storedFilename` 或文件 ID。
- 在服务层增加统一的 `resolveUploadedFileOnly()`，只解析 `uploads` 目录下的随机文件名。
- 禁止 `..`、绝对路径、盘符路径、UNC 路径和任意相对路径。
- OCR API 内部函数也要做防御，不能只依赖路由层校验。

### 2. 业务模型新增、修改、删除缺少管理员权限

严重程度：高

位置：

- `vlmOcr_api/app.js:27`
- `vlmOcr_api/routes/models.js:153`
- `vlmOcr_api/routes/models.js:174`
- `vlmOcr_api/routes/models.js:190`

问题说明：

`app.js` 对 `/api/models` 整体加了 `authMiddleware`，但在 `routes/models.js` 里只有视觉模型供应商的写接口使用了 `requireAdmin`。业务模型本身的 `POST /api/models`、`PUT /api/models/:id`、`DELETE /api/models/:id` 只要求登录，不要求管理员。

影响：

- 普通用户可以新增、篡改或删除全局业务模型。
- 可通过修改 `glmTips`、关键词、`moreApi` 影响所有用户的 OCR 分类结果。
- 删除默认模型会导致识别流程异常或结果不可用。

建议：

- 给 `router.post('/')`、`router.put('/:id')`、`router.delete('/:id')` 增加 `requireAdmin`。
- 如果未来需要用户私有模型，应新增 `owner_user_id` 并按资源归属隔离，不要混用全局模型表。

### 3. 上传文件没有用户归属，登录用户可枚举和删除所有上传文件

严重程度：高

位置：

- `vlmOcr_api/routes/upload.js:56`
- `vlmOcr_api/routes/upload.js:77`
- `vlmOcr_api/routes/upload.js:90`
- `vlmOcr_api/routes/upload.js:105`
- `vlmOcr_api/routes/upload.js:125`
- `vlmOcr_api/app.js:35`

问题说明：

上传文件保存到统一的 `vlmOcr_api/uploads` 目录，只通过随机文件名区分，没有记录 `user_id` 或文件归属。`GET /api/upload/list` 会列出目录下全部文件，`DELETE /api/upload/:filename` 可删除任意合法文件名。虽然 `/api/upload` 本身需要登录，但没有做资源归属校验。

此外，`/api/upload/files/:filename` 被挂载为公开下载路由，不需要认证。只要文件 URL 泄露，任何人都能访问。

影响：

- 任意登录用户可查看其他用户上传文件列表。
- 任意登录用户可删除其他用户上传文件。
- 文件 URL 一旦被分享、日志记录或被猜到，就可被未登录用户下载。

建议：

- 新建上传文件元数据表，记录 `stored_filename`、原始文件名、大小、MIME、上传用户、创建时间。
- 列表、删除、下载接口按 `req.user.userId` 校验归属；管理员可额外查看全部。
- 若必须公开访问，应生成短期签名 URL，而不是长期裸露文件名。
- OCR 历史记录保存文件 ID，而不是仅保存公开 URL。

## P1 / 中高风险问题

### 4. GLM 请求的临时文件清理时机可能过早

严重程度：中高

位置：

- `vlmOcr_api/ocrApi/GLMFlash.js:142`
- `vlmOcr_api/ocrApi/GLMFlash.js:188`

问题说明：

`callGLMChat()` 在构造请求体前会读取图片为 base64，因此本地图片文件本身可以提前清理。但 `req.end(() => { imageContent.cleanup(); })` 的回调只表示请求数据已交给底层系统，并不表示远端响应已经完成。当前清理逻辑和 Promise resolve/reject 生命周期分离，未来如果 `imageContent` 中增加流式资源、调试文件或其他延迟使用对象，会出现竞态。

影响：

- 当前 base64 场景风险有限，但清理职责位置不稳。
- 请求过程中如果发生异常，清理路径依赖多个分支，后续维护容易引入临时文件泄漏或提前删除。

建议：

- 把 `cleanup()` 放到 Promise 完成后的统一 `finally` 风格逻辑中。
- 或改为 `try { await request } finally { cleanup() }` 的 async/await 结构，避免清理和网络生命周期脱节。

### 5. 历史记录搜索模式分页总数不准确

严重程度：中

位置：

- `vlmOcr_api/routes/history.js:24`
- `vlmOcr_api/routes/history.js:25`
- `vlmOcr_api/routes/history.js:26`
- `vlmOcr_api/services/databaseService.js:248`

问题说明：

当请求 `/api/history?search=xxx&page=2&limit=10` 时，路由调用 `DatabaseService.searchImages(search, limitNum, req.user)`，该服务只接收 `limit`，没有 `offset`。随后 `total = data.length`。

影响：

- 搜索结果永远返回第一页。
- `total` 只是当前页条数，不是真实匹配总数。
- 前端分页的 `totalPages` 会错误，用户无法浏览全部搜索结果。

建议：

- `searchImages(searchTerm, limit, offset, actor)` 增加 offset。
- 增加 `countImages({ search })` 或单独的 `countSearchImages()`。
- 搜索分支和普通列表分支使用同一套分页响应逻辑。

### 6. 统计日期校验允许反向日期，`summary` 未限制日期范围

严重程度：中

位置：

- `vlmOcr_api/routes/statistics.js:13`
- `vlmOcr_api/routes/statistics.js:24`
- `vlmOcr_api/routes/statistics.js:32`
- `vlmOcr_api/routes/statistics.js:149`
- `vlmOcr_api/routes/statistics.js:155`

问题说明：

`validateDateRange()` 使用 `Math.abs(end - start)` 判断范围，没有拒绝 `startDate > endDate`。随后 `generateDateSeries(start, end)` 在起始日期大于结束日期时会返回空数组。

同时 `/api/statistics/summary` 直接使用 `req.query.startDate` 和 `req.query.endDate`，没有调用 `validateDateRange()`。用户可以传入超大日期范围，触发大量日期序列生成和数据库日期函数查询。

影响：

- 日期反向时接口返回空数据但不报错，容易误导前端。
- 超大范围可能造成慢查询和内存开销。
- 无效日期字符串可能让统计结果异常。

建议：

- 明确校验 `start <= end`。
- 所有统计接口统一调用日期校验函数，包括 `summary` 和 `model-ocr`。
- 限制最大范围，例如 31 天或 90 天。
- SQL 中尽量避免 `DATE_FORMAT(created_at)` 包裹字段，可改成范围查询以使用索引。

### 7. `ocrInfo` 对象直接写入 TEXT 字段，读取历史时类型不稳定

严重程度：中

位置：

- `vlmOcr_api/services/databaseService.js:105`
- `vlmOcr_api/routes/ocr.js:67`
- `vlmOcr_api/routes/classifyOcr.js:78`
- `vlmOcr_api/routes/ocr.js:28`

问题说明：

`img_history.ocrInfo` 是 `TEXT` 字段，但 `saveImageHistory()` 直接把 `data.ocrInfo` 赋给数据库记录。调用方经常传入对象，例如 Paddle 返回的 `response.json` 或 `modelRes.ocrInfo`。mysql2 会如何序列化对象取决于驱动处理，常见结果可能是 `"[object Object]"`，也可能报错或转成不可预期内容。

读取历史时，`normalizeHistoryResponse()` 又把 `historyRecord.ocrInfo || {}` 作为 `result` 返回，没有 JSON.parse。这样新识别返回对象，历史返回字符串，前端接收到的数据结构不一致。

影响：

- 历史记录中的结构化 OCR 结果可能损坏。
- 同一接口新识别和命中历史返回类型不一致。
- 前端渲染和导出逻辑容易出现运行时错误。

建议：

- 写入时统一 `JSON.stringify()`。
- 读取时统一 `safeJsonParse()`，解析失败则返回 `{ text: raw }` 或保留字符串字段。
- 数据库字段可改为 MySQL `JSON` 类型，或继续用 TEXT 但明确序列化契约。

### 8. `/classify` 路由配置了文件上传但实际不使用上传文件

严重程度：中

位置：

- `vlmOcr_api/routes/classify.js:3`
- `vlmOcr_api/routes/classify.js:6`
- `vlmOcr_api/routes/classify.js:12`
- `vlmOcr_api/routes/classify.js:15`
- `vlmOcr_api/routes/classify.js:19`

问题说明：

`/classify` 使用 `upload.single('image')` 接收文件，`multer({ dest: 'uploads/' })` 会先把文件落盘。但路由随后只检查 `req.body.imageUrl`，完全不使用 `req.file`，也没有清理上传文件。

影响：

- 用户上传文件但不传 `imageUrl` 时，接口返回 400，但文件已经留在 `uploads/`。
- 可被恶意请求刷满磁盘。
- 该路由的上传配置没有使用统一的文件安全策略。

建议：

- 如果 `/classify` 只支持 URL，移除 `multer`。
- 如果要支持文件上传，使用 `routes/upload.js` 中的统一校验和随机文件名策略，并在失败时清理临时文件。
- 在 `finally` 中处理失败请求的文件清理。

### 9. 视觉模型配置停用后仍可能被已有业务模型调用

严重程度：中

位置：

- `vlmOcr_api/routes/classifyOcr.js:61`
- `vlmOcr_api/routes/classifyOcr.js:63`
- `vlmOcr_api/services/databaseService.js:164`
- `vlmOcr_api/services/databaseService.js:210`

问题说明：

创建和更新业务模型时会通过 `assertEnabledVisionProvider()` 禁止选择停用供应商。但实际识别时，`classifyOcr` 使用 `getVisionModelProviderByValue()` 获取供应商后直接调用，没有检查 `provider.enabled`。

影响：

- 管理员停用供应商后，已有模型仍然可以继续调用该供应商。
- “停用”配置和运行时行为不一致。

建议：

- `classifyOcr` 运行时改用 `assertEnabledVisionProvider(bestModel.moreApi)`。
- 当供应商停用时返回明确错误，或让模型自动降级到可用供应商。

### 10. 上传接口仅信任扩展名和客户端 MIME，未校验真实文件内容

严重程度：中

位置：

- `vlmOcr_api/utils/fileSecurity.js:24`
- `vlmOcr_api/utils/fileSecurity.js:28`
- `vlmOcr_api/routes/upload.js:43`

问题说明：

`isAllowedFile()` 只检查原始文件名扩展名和客户端传入的 `mimetype`。这两个值都可以伪造。攻击者可以上传伪装成 JPG/PDF 的其他内容。

影响：

- 可能造成恶意文件托管。
- 下游 OCR/PDF 转换库处理伪造文件时可能出现异常或资源消耗。
- 公开下载路由会把伪造文件分发给客户端。

建议：

- 使用文件头魔数校验真实类型，例如 JPG/PNG/WEBP/PDF 签名。
- 对 PDF 做页数、大小和解析超时限制。
- 下载接口设置安全响应头，例如 `X-Content-Type-Options: nosniff`。

## P2 / 中低风险与质量问题

### 11. `form-data` 被代码直接引用，但未声明为直接依赖

严重程度：中低

位置：

- `vlmOcr_api/ocrApi/PaddleOcrVl.js:5`
- `vlmOcr_api/package.json:11`
- `vlmOcr_api/package-lock.json:10`

问题说明：

`PaddleOcrVl.js` 直接 `require('form-data')`，但 `package.json` 的根依赖没有声明 `form-data`。当前本机能解析到它，是因为其他依赖把它安装到了 `node_modules`，属于依赖提升带来的偶然可用。

影响：

- 换 npm 版本、包管理器或安装策略后可能出现 `Cannot find module 'form-data'`。
- 依赖关系不清晰，部署可重复性变差。

建议：

- 在 `vlmOcr_api/package.json` 中显式加入 `form-data`。
- 后续清理 `crypto`、`path`、`vue`、`vue-router` 等后端不需要或内置依赖。

### 12. `deleteImageHistory()` 先按权限查记录，再只按 ID 删除，存在维护隐患

严重程度：中低

位置：

- `vlmOcr_api/services/databaseService.js:115`
- `vlmOcr_api/services/databaseService.js:117`

问题说明：

`deleteImageHistory(id, actor)` 先调用 `getImageById(id, actor)` 做权限过滤，如果查到记录，再执行 `MySQLHelper.delete('img_history', { id })`。当前因为 `id` 是主键，实际删除的是同一条记录，风险不高。但权限条件没有体现在最终 DELETE SQL 中，未来如果删除逻辑改动或主键条件变化，容易变成越权删除。

建议：

- 最终 DELETE 语句也带上用户范围条件。
- 对 `deleteImageByMd5()` 和 `deleteImageHistory()` 统一封装带 scope 的删除方法。

### 13. 管理员创建默认密码仍有弱口令回退

严重程度：中低

位置：

- `vlmOcr_api/services/schemaService.js:48`

问题说明：

`ensureDefaultAdmin()` 在未设置 `DEFAULT_ADMIN_PASSWORD` 时使用 `123456` 作为默认管理员密码。虽然首次启动方便，但生产或测试环境很容易遗留弱口令。

影响：

- 如果部署后没有立即修改密码，管理员账号容易被撞库或猜测。

建议：

- 生产环境要求必须设置 `DEFAULT_ADMIN_PASSWORD`。
- 或首次启动生成随机密码并只输出一次到安全日志。
- 登录后强制管理员修改默认密码。

### 14. SQL 迁移脚本非幂等，重复执行会失败

严重程度：低

位置：

- `vlmOcr_api/sql/20260606_p0_p1_security_migration.sql`

问题说明：

迁移脚本直接 `ADD COLUMN role`、`ADD COLUMN user_id` 和 `CREATE INDEX`，没有检查字段或索引是否已存在。重复执行会报错。虽然 `schemaService.js` 里有幂等逻辑，但单独执行 SQL 文件时仍会失败。

建议：

- MySQL 8 可使用 `ADD COLUMN IF NOT EXISTS`。
- 或改为存储过程/信息模式检查后再执行。
- 文档中明确该脚本只能执行一次，推荐使用启动时 schema 初始化。

### 15. 遗留 `ppOcr.js` 存在命令拼接和任意 URL 下载风险

严重程度：低到中，取决于是否仍会被引用

位置：

- `vlmOcr_api/utils/ppOcr.js:4`
- `vlmOcr_api/utils/ppOcr.js:142`
- `vlmOcr_api/utils/ppOcr.js:143`
- `vlmOcr_api/utils/ppOcr.js:239`
- `vlmOcr_api/utils/ppOcr.js:273`
- `vlmOcr_api/utils/ppOcr.js:274`

问题说明：

当前路由引用的是 `utils/classifier.js`，没有看到生产路由直接引用 `ppOcr.js`。但这个遗留文件仍在仓库中，里面有 `execSync(command)` 字符串拼接执行 Python，以及不带白名单、大小限制、超时和内网防护的 URL 下载。

影响：

- 如果未来误引用该文件，会重新引入命令注入、SSRF、大文件下载和路径穿越风险。
- 安全审计和维护时容易混淆当前实现。

- 如果已废弃，删除 `ppOcr.js` 或移动到明确的归档目录。
- 如果仍需要，改为 `spawnFile`/`execFile` 参数数组调用，并复用 `fileSecurity.downloadAllowedUrl()`。

## 已观察到的正向改进

- `config/appConfig.js` 对 `JWT_SECRET`、数据库配置、生产 CORS 白名单做了启动必填校验。
- `middleware/auth.js` 每次请求会重新从数据库读取用户，避免单靠 token 中的角色字段做权限判断。
- `utils/mysql.js` 大部分查询已经使用参数化，分页 limit/offset 也做了整数化处理。
- `utils/fileSecurity.js` 对远程 HTTPS URL 做了域名白名单、DNS 解析、内网 IP 拦截、大小限制和禁用重定向。
- `databaseService.js` 已经为历史记录和统计加入用户范围过滤。

1. 立即禁止外部接口传入任意本地文件路径，只允许上传目录中的受控文件 ID。

3. 为上传文件增加用户归属，列表、删除、下载都按归属校验。
4. 修复历史搜索分页总数和 offset。
5. 统一 `ocrInfo` 的 JSON 序列化与反序列化。
6. 统一统计日期校验，限制 `summary` 查询范围。
7. 移除或修复 `/classify` 中未使用的 multer 上传。
8. 显式声明 `form-data` 依赖，清理后端无关依赖。
9. 删除或隔离遗留 `ppOcr.js`。
